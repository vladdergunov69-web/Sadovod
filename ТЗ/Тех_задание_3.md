## 1. ВВЕДЕНИЕ

### 1.1. Наименование программы

Наименование — Веб-приложение интернет-магазина «Садовод». Настоящее дополнение № 3 описывает три самостоятельные доработки: автоматизацию развёртывания, модуль «Избранное» и систему аудита входа пользователей.

### 1.2. Краткая характеристика области применения программы

Программа предназначена для организации розничной онлайн-торговли садово-огородными товарами. Настоящее дополнение расширяет эксплуатационные и функциональные возможности приложения тремя независимыми друг от друга доработками:

- **Автоматизация развёртывания** — упрощает установку приложения и базы данных на сервере администратора до выполнения одной команды средствами Docker Compose, вместо ручной многошаговой настройки.
- **Модуль «Избранное»** — даёт покупателю возможность отмечать товары каталога для последующего быстрого доступа к ним, не используя для этого корзину; добавлена по заявке постоянного покупателя магазина.
- **Система аудита входа пользователей** — фиксирует каждую попытку входа (успешную и неуспешную) и время выхода из системы, что повышает защищённость учётных записей и позволяет администратору выявлять попытки подбора пароля.

Доработки независимы друг от друга и могут внедряться по отдельности; в настоящем документе они объединены в одно техническое задание как работы одного этапа учебной практики (08.06.2026–19.06.2026).

## 3. НАЗНАЧЕНИЕ РАЗРАБОТКИ

### 3.1. Функциональное назначение программы

Доработка обеспечивает следующий функционал, разделённый по трём направлениям.

#### 3.1.1. Доработка 1. Автоматизация развёртывания

1. Подготовка файла `Dockerfile` для сборки образа приложения на основе официального образа .NET SDK (этап сборки) и ASP.NET Core Runtime (этап выполнения) — многоэтапная сборка (multi-stage build); образ не содержит исходный код и инструменты сборки в финальном слое.
2. Контейнеризация базы данных на основе официального образа Microsoft SQL Server с автоматическим выполнением скрипта инициализации `Scripts/InitDatabase.sql` при первом запуске (создание схемы, триггеров аудита, хранимой процедуры и начальных данных).
3. Единый файл `docker-compose.yml` с сервисами `web` и `db`, их сетевым взаимодействием, именованными томами для данных СУБД и изображений товаров, переменными окружения и проверкой готовности базы данных (healthcheck) перед запуском приложения.
4. Файл `.env.example` с шаблоном переменных окружения (пароль администратора SQL Server, строка подключения, ключи VAPID, порт публикации сайта); реальный файл `.env` в репозиторий не включается.
5. Развёртывание полного стенда выполняется единственной командой `docker compose up -d --build`.
6. Изменения структуры базы данных не вносятся — используется существующий скрипт `Scripts/InitDatabase.sql` без модификации схемы.

#### 3.1.2. Доработка 2. Модуль «Избранное» (по заявке пользователя)

1. На карточке товара в каталоге и на странице товара отображается значок «В избранное»; состояние значка отражает, отмечен ли товар у текущего покупателя.
2. Нажатие на значок добавляет товар в «Избранное» без перезагрузки страницы (AJAX-запрос); повторное нажатие убирает товар из «Избранного».
3. Действие доступно только авторизованным покупателям; гостю при нажатии предлагается войти в систему.
4. Новый пункт «Избранное» в выпадающем меню учётной записи с указанием количества отмеченных товаров.
5. Страница «Избранное» (`/Favorites`) отображает все отмеченные товары в виде карточек; с карточки можно убрать товар из списка либо сразу добавить его в корзину.
6. Если ранее отмеченный товар удалён администратором, он автоматически перестаёт отображаться в «Избранном» (каскадное удаление связи).
7. Доработка добавляет новую таблицу `Favorites` (`UserId`, `ProductId`, `AddedAt`) с уникальным составным ограничением (`UserId`, `ProductId`) и внешними ключами с каскадным удалением на `Users` и `Products`; прочие таблицы не изменяются.

#### 3.1.3. Доработка 3. Система аудита входа пользователей

1. При каждой попытке входа в систему (форма `/Account/Login`) создаётся запись в новом журнале `LoginAuditLog`: введённый логин, результат попытки (Success/Failed), IP-адрес, дата и время.
2. При успешном входе в запись сохраняется идентификатор пользователя; при неуспешном входе — только если введённый логин существует в системе.
3. При выходе пользователя (`Account/Logout`) в соответствующей записи журнала фиксируется время выхода (`LogoutAt`).
4. Новый раздел административной панели «Аудит входов» (`/Admin/LoginAudit`, роль «Администратор»): таблица записей, фильтры по логину, результату и диапазону дат, постраничная навигация (25 записей на страницу).
5. Визуальное выделение логина, по которому за последние 15 минут зафиксировано 5 и более неуспешных попыток подряд, — индикатор подозрительной активности.
6. В журнал никогда не записывается пароль (введённый или хэшированный) — только логин и результат попытки.
7. Доработка добавляет новую таблицу `LoginAuditLog` (`Login`, `UserId` — необязательно, `Result`, `IpAddress`, `LoginAt`, `LogoutAt`) с внешним ключом на `Users` (`ON DELETE SET NULL`); прочие таблицы не изменяются.

#### 3.1.4. Общесистемные требования к реализации

- Все три доработки реализуются в составе существующего приложения «Садовод» на платформе ASP.NET Core MVC (.NET 10) с базой данных Microsoft SQL Server, без изменения архитектурного шаблона и без затрагивания существующего функционала.
- Доработки независимы друг от друга: автоматизация развёртывания не зависит от состава таблиц базы данных и одинаково упаковывает приложение независимо от наличия модулей «Избранное» и «Аудит входов».
- Изменения структуры базы данных ограничены добавлением двух новых таблиц (`Favorites`, `LoginAuditLog`); ни одна из существующих таблиц не изменяется.
- Все новые изменяющие операции (добавление/удаление избранного) защищены CSRF-токеном, как и прочие формы приложения.

|---|---|
| Посетитель (гость) | Видит значок «В избранное» на карточках, но при нажатии перенаправляется на страницу входа. Каждая его попытка входа в систему (успешная и неуспешная) фиксируется в журнале аудита входов. |
| Покупатель (авторизованный) | Отмечает и снимает отметку товаров как избранные; просматривает страницу «Избранное»; добавляет избранные товары в корзину. Его входы и выходы фиксируются в журнале аудита. |
| Администратор | Развёртывает приложение и базу данных командой `docker compose up -d --build`; просматривает раздел «Аудит входов» с фильтрами и индикатором подозрительной активности; при удалении товара его записи в `Favorites` удаляются автоматически. |

Доработанное приложение предназначено для эксплуатации на автоматизированном рабочем месте администратора (ПК, ноутбук, сервер с установленным Docker) и на устройствах покупателей — ПК, ноутбуках, смартфонах, планшетах — с установленным современным браузером.

Структура хранимых данных, затронутая доработкой 2 «Избранное»:

| Таблица | Ключевые поля и ограничения |
|---|---|

Прочие таблицы базы данных (`Products`, `Users`, `Orders` и др.) используются в исходном виде. Операции добавления/удаления записей в `Favorites` доступны только владельцу записи — покупателю, выполнившему вход.

Структура хранимых данных, затронутая доработкой 3 «Аудит входов»:

| Таблица | Ключевые поля и ограничения |
|---|---|
| `LoginAuditLog` (новая) | `Login` до 100 символов, обязательно; `UserId` (внешний ключ на `Users`, `ON DELETE SET NULL`, необязательно); `Result` — 1 символ (S/F: Success/Failed); `IpAddress` до 45 символов; `LoginAt` по умолчанию `GETDATE()`; `LogoutAt` — необязательно; индекс по `Login` и по `LoginAt`. |

Просмотр и фильтрация журнала `LoginAuditLog` доступны только пользователю с ролью «Администратор» (политика `AdminOnly`).

#### 4.1.6. Требования к организации входных данных

Входные данные поступают:

1. по доработке 1: через файл `.env` (пароль СУБД, строка подключения, ключи VAPID, порт сайта) и команды интерфейса командной строки Docker Compose;
2. по доработке 2: через нажатие значка «В избранное» (AJAX-запросы `POST /Favorites/Add/{productId}`, `POST /Favorites/Remove/{productId}`) и переход по ссылке `/Favorites`;
3. по доработке 3: через форму входа `/Account/Login` (логин, пароль — пароль не журналируется), HTTP-запрос (IP-адрес отправителя), действие `/Account/Logout`, параметры фильтра в разделе «Аудит входов».

На стороне сервера выполняется валидация: для доработки 2 — проверка авторизации, существования товара, верификация CSRF-токена, проверка отсутствия дублирующей записи; для доработки 3 — проверка прав доступа к разделу аудита и корректности параметров фильтра.

#### 4.1.7. Требования к организации выходных данных

1. Готовое к работе веб-приложение по адресу `http://localhost:8080` после выполнения команды `docker compose up -d --build`; журнал выполнения с информацией о состоянии контейнеров.
2. Обновлённое состояние значка «В избранное» без перезагрузки страницы; страница `/Favorites` со списком отмеченных товаров; счётчик в меню учётной записи.
3. Страница раздела «Аудит входов»: таблица записей с визуальным выделением неуспешных попыток и индикатором подозрительной активности.

#### 4.1.8. Требования к временным характеристикам

- Полное развёртывание стенда «с нуля» — не более 10 минут; повторный запуск без пересборки — не более 30 секунд (доработка 1).
- Отклик на нажатие значка «В избранное» — не более 1 секунды; загрузка страницы «Избранное» — не более 2–3 секунд (доработка 2).
- Дополнительное время на запись события входа — не более 200 мс; формирование страницы журнала аудита при объёме до нескольких тысяч записей — до 3 секунд (доработка 3).

### 4.2. Требования к надёжности

Надёжность обеспечивается:

- настройкой проверки готовности контейнера базы данных (healthcheck) и политикой перезапуска контейнеров при сбое (доработка 1);
- уникальным составным ограничением (`UserId`, `ProductId`) и каскадным удалением записей `Favorites` при удалении пользователя или товара (доработка 2);
- записью события входа независимо от результата аутентификации и сохранением истории входов при удалении учётной записи (`ON DELETE SET NULL`) (доработка 3);
- устойчивостью записи лог-события аудита входа к ошибкам — сбой записи не должен препятствовать самому входу пользователя в систему.

- периодическое резервное копирование тома данных SQL Server и базы данных в целом (рекомендация);

- назначение ответственного за хранение файла `.env` и пароля администратора СУБД.

#### 4.2.10. Время восстановления после отказа

После сбоя контейнера приложения или базы данных восстановление выполняется автоматическим перезапуском контейнера либо командой `docker compose restart` — не более 1 минуты. Отказ модуля «Избранное» или системы аудита входов не влияет на работу остального приложения — это независимые функциональные блоки, восстанавливаемые вместе с базой данных.

#### 4.2.11. Отказы из-за некорректных действий оператора

Операции по доработкам 2 и 3 ограничены ролевой моделью доступа (покупатель — только собственные записи избранного; администратор — только просмотр журнала аудита) и защищены CSRF-токеном.

## 8. ПОРЯДОК КОНТРОЛЯ И ПРИЁМКИ

### 8.1. Виды испытаний

Контроль и приёмка осуществляются руководителем практики / преподавателем по результатам представления материалов и контрольной демонстрации по каждой из трёх доработок.

**Комплект материалов:**

- файлы `Dockerfile`, `docker-compose.yml`, `.env.example`, `.dockerignore` (в составе репозитория);
- исходный код доработок «Избранное» и «Аудит входов»;
- настоящее техническое задание и заявка пользователя;
- три инструкции со скриншотами; отчёт и дневник по практике.

**Контрольная демонстрация. Доработка 1 — автоматизация развёртывания:**

1. На чистой машине (или после `docker compose down -v`) выполняется команда `docker compose up -d --build`.
2. Проверяется автоматический запуск контейнера базы данных и выполнение скрипта инициализации.
3. Проверяется доступность сайта по адресу `http://localhost:8080` и вход под учётной записью администратора.
4. Стенд останавливается (`docker compose down` без `-v`) и запускается повторно — проверяется сохранность данных.

**Контрольная демонстрация. Доработка 2 — модуль «Избранное»:**

1. Вход под покупателем; на карточке товара нажимается значок «В избранное» — значок меняет состояние без перезагрузки страницы.
2. Переход в раздел «Избранное» — отображается отмеченный товар, счётчик в меню соответствует количеству товаров.
3. Попытка нажать значок без входа в систему — выполняется переход на страницу входа.
4. Администратор удаляет товар, ранее отмеченный покупателем, — товар автоматически пропадает из «Избранного».

**Контрольная демонстрация. Доработка 3 — аудит входов:**

1. Успешный и неуспешный вход — в разделе «Аудит входов» появляются соответствующие записи с верными данными.
2. Выход пользователя — в записи журнала заполняется время выхода.
3. Фильтрация журнала по логину, результату и диапазону дат — список обновляется согласно условиям.
4. Пять и более неуспешных попыток подряд по одному логину за 15 минут — записи визуально выделяются как подозрительная активность.
