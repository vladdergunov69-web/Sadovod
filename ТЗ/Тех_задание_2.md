## 1. ИСПРАВЛЕНИЕ ДЕФЕКТА: НЕ РАБОТАЛО ОФОРМЛЕНИЕ ЗАКАЗА

### 1.1. Описание дефекта

При оформлении заказа на странице `/Orders/Create` после нажатия кнопки «Подтвердить заказ» заказ не создавался: пользователь получал сообщение «Не удалось создать заказ. Попробуйте ещё раз.». Оформить заказ было невозможно ни одному покупателю.

### 1.2. Причина

Контекст данных `SadovodDbContext` перед каждым сохранением вызывает хранимую процедуру `dbo.SetSessionUserContext` (для передачи идентификатора текущего пользователя триггерам аудита). Вызов выполняется через низкоуровневую ADO-команду на соединении с БД.

Оформление заказа в `OrdersController.Create` обёрнуто в явную транзакцию (`BeginTransactionAsync()`), чтобы атомарно списать остатки и создать заказ. При наличии активной транзакции Microsoft SQL Server требует, чтобы каждая команда на соединении была **явно привязана к этой транзакции**. Команда вызова процедуры такой привязки не имела, поэтому при сохранении заказа возникало исключение, которое перехватывалось блоком `catch`, транзакция откатывалась, а пользователь видел сообщение об ошибке.

Прочие операции (создание/изменение/удаление товара, редактирование профиля) выполнялись вне явной транзакции, поэтому работали корректно — дефект проявлялся только при оформлении заказа.

### 1.3. Внесённое исправление

В метод `SetSessionUserContextAsync` контекста `SadovodDbContext` добавлена привязка ADO-команды к текущей транзакции контекста, если она открыта:

```csharp
var currentTx = Database.CurrentTransaction?.GetDbTransaction();
if (currentTx is not null)
    cmd.Transaction = currentTx;
```

Вне транзакции (значение `CurrentTransaction` равно `null`) поведение не изменилось — ранее работавший функционал не затронут.

**Файл:** `Data/SadovodDbContext.cs`.

### 1.4. Критерий приёмки

Авторизованный покупатель с непустой корзиной успешно оформляет заказ: создаётся заказ со статусом «Новый», уменьшаются остатки товаров на складе, корзина очищается, заказ отображается в разделе «Мои заказы».

## 2. ДОРАБОТКА 1. УПРАВЛЕНИЕ ЗАКАЗАМИ В АДМИНИСТРАТИВНОЙ ПАНЕЛИ

### 2.1. Обоснование

В первоначальном техническом задании управление заказами было заявлено (раздел 3.1, п. 17), однако в административной панели **отсутствовала вкладка «Заказы»** и соответствующий функционал. Доработка закрывает этот пробел: в административную панель добавлена вкладка **«Заказы»**.

Изменения структуры базы данных не требуются — используются существующие таблицы `Orders`, `OrderItems`, `OrderStatuses`, `Users`.

### 2.2. Функциональные требования

1. В навигационном меню административной панели добавлен пункт **«Заказы»** (`/Admin/Orders`).
2. Раздел доступен только пользователям с ролью «Администратор» (политика `AdminOnly`).
3. Отображается список всех заказов магазина со следующими данными: номер заказа, дата и время оформления, покупатель (ФИО и логин), состав заказа (наименования товаров и количество), итоговая сумма, текущий статус.
4. Список отсортирован по дате оформления по убыванию (новые заказы — сверху) и разбит на страницы (по 15 заказов на страницу).
5. Предусмотрен фильтр по статусу заказа (Новый / Обработан / Выполнен / Отменён / Все).
6. Администратор может изменить статус любого заказа выбором значения из выпадающего списка статусов с подтверждением кнопкой. Изменение статуса защищено CSRF-токеном.
7. При изменении статуса заказа покупателю, оформившему заказ, автоматически отправляется адресное Web Push-уведомление с новым статусом — при наличии у него активной подписки на уведомления. Если подписки нет, заказ всё равно обновляется.

### 2.3. Реализация

- Новый контроллер `Areas/Admin/Controllers/OrdersController.cs` — действия `Index` (список с фильтром и пагинацией) и `ChangeStatus` (смена статуса, POST).
- Новая модель представления `Models/ViewModels/AdminOrderListViewModel.cs`.
- Новое представление `Areas/Admin/Views/Orders/Index.cshtml`.
- В сервис уведомлений `Services/PushNotificationService.cs` добавлен метод адресной отправки подписчику `SendToUserAsync(int? userId, …)`; общая логика рассылки вынесена в приватный метод и переиспользуется методами `SendToAllAsync` и `SendToUserAsync`.
- В макет административной панели `Areas/Admin/Views/Shared/_AdminLayout.cshtml` добавлен пункт меню «Заказы».

### 2.4. Критерий приёмки

Администратор видит список всех заказов, фильтрует его по статусу, изменяет статус заказа; покупатель видит обновлённый статус в разделе «Мои заказы» и (при наличии подписки) получает push-уведомление.

## 3. ДОРАБОТКА 2. РАСШИРЕННЫЕ ФИЛЬТРЫ И СОРТИРОВКА КАТАЛОГА

### 3.1. Обоснование

Расширение возможностей публичного каталога товаров для удобства подбора товаров покупателем. Изменения структуры базы данных не требуются — используется существующая таблица `Products`.

### 3.2. Функциональные требования

В дополнение к существующим текстовому поиску по наименованию и фильтрации по категории на странице каталога (`/Products`) реализованы:

1. Фильтр по диапазону цены — поля «Цена от» и «Цена до» (оба необязательны, могут использоваться по отдельности).
2. Фильтр «Только в наличии» — отображение товаров с положительным остатком на складе (`Quantity > 0`).
3. Сортировка результатов:
   - по наименованию (А→Я) — по умолчанию;
   - по цене по возрастанию;
   - по цене по убыванию;
   - по остатку на складе по убыванию.
4. Все выбранные параметры фильтрации и сортировки сохраняются при переходе по страницам пагинации; кнопка «Сброс» возвращает каталог к исходному виду.

### 3.3. Реализация

- В модель представления `Models/ViewModels/ProductListViewModel.cs` добавлены свойства `MinPrice`, `MaxPrice`, `InStockOnly`, `Sort`.
- В контроллере `Controllers/ProductsController.cs` (действие `Index`) к запросу `IQueryable<Product>` динамически достраиваются условия `Where` (диапазон цены, наличие) и выбирается порядок `OrderBy`/`OrderByDescending` в зависимости от параметра сортировки. Запрос выполняется к БД единожды (отложенное выполнение Entity Framework Core).
- В представлении `Views/Products/Index.cshtml` добавлены поля диапазона цены, флажок наличия, выпадающий список сортировки; параметры проброшены в ссылки постраничной навигации.

### 3.4. Критерий приёмки

Покупатель ограничивает выдачу каталога по цене и наличию, меняет порядок сортировки; выбранные параметры сохраняются при листании страниц.

## 4. ДОРАБОТКА 3. УПРАВЛЕНИЕ ПОЛЬЗОВАТЕЛЯМИ И РОЛЯМИ

### 4.1. Обоснование

Предоставление администратору возможности просматривать учётные записи и изменять роль пользователя (Администратор ↔ Пользователь). Изменения структуры базы данных не требуются — используются существующие таблицы `Users` и `Roles`.

### 4.2. Функциональные требования

1. В навигационном меню административной панели добавлен пункт **«Пользователи»** (`/Admin/Users`).
2. Раздел доступен только пользователям с ролью «Администратор» (политика `AdminOnly`).
3. Отображается список всех пользователей: идентификатор, логин (email), ФИО, телефон, текущая роль. Список разбит на страницы (по 20 пользователей).
4. Администратор может изменить роль пользователя выбором значения из выпадающего списка с подтверждением кнопкой. Операция защищена CSRF-токеном.
5. **Защита от блокировки доступа:** администратор не может изменить роль собственной учётной записи (предотвращение случайной потери прав доступа). Для собственной строки элементы смены роли скрыты.
6. Изменение роли фиксируется в журнале аудита автоматически (см. п. 4.3).

### 4.3. Доработка триггера аудита

Существующий триггер `trg_Users_Update` реагировал только на изменение полей ФИО и телефона. Для аудита смены роли триггер **расширен** (изменён существующий объект БД; новые таблицы и столбцы не добавлялись):

- триггер срабатывает дополнительно при изменении поля `RoleId`;
- в записи журнала аудита (поля `OldValue`/`NewValue`, формат JSON) добавлено значение `RoleId` до и после изменения.

Определение триггера в `Scripts/InitDatabase.sql` переписано в форме `CREATE OR ALTER TRIGGER`, что позволяет безопасно повторно применить скрипт для обновления триггера на действующей базе данных.

### 4.4. Реализация

- Новый контроллер `Areas/Admin/Controllers/UsersController.cs` — действия `Index` (список с пагинацией) и `ChangeRole` (смена роли, POST, с защитой от смены собственной роли).
- Новая модель представления `Models/ViewModels/UserListViewModel.cs`.
- Новое представление `Areas/Admin/Views/Users/Index.cshtml`.
- Доработка триггера `trg_Users_Update` в `Scripts/InitDatabase.sql`.
- В макет административной панели `Areas/Admin/Views/Shared/_AdminLayout.cshtml` добавлен пункт меню «Пользователи».

### 4.5. Критерий приёмки

Администратор просматривает список пользователей и изменяет роль выбранного пользователя; собственную роль изменить невозможно; смена роли (после применения обновлённого триггера) отображается в журнале аудита с указанием старого и нового значения `RoleId`.

## 6. ПОРЯДОК ПРИМЕНЕНИЯ ДОРАБОТОК

1. Обновить исходный код проекта и выполнить сборку:
   ```powershell
   dotnet build
   ```
2. (Только для доработки 3 — аудит смены роли) Применить обновлённый триггер на действующей базе данных, повторно выполнив скрипт инициализации либо отдельный фрагмент с триггером `trg_Users_Update`:
   ```powershell
   sqlcmd -S .\SQLEXPRESS -E -d SadovodShop -i "Scripts\InitDatabase.sql"
   ```
   Скрипт идемпотентен; триггер обновляется через `CREATE OR ALTER`, прочие объекты и данные не затрагиваются.
3. Запустить приложение:
   ```powershell
   dotnet run
   ```

> Примечание. Для работы адресных push-уведомлений (доработка 1) должны быть настроены VAPID-ключи в `appsettings.json`, а покупатель должен быть подписан на уведомления. При отсутствии подписки смена статуса заказа выполняется штатно, без отправки уведомления.

## 7. ПОРЯДОК КОНТРОЛЯ И ПРИЁМКИ ДОРАБОТОК

Контрольная демонстрация:

1. **Оформление заказа (исправление дефекта):** войти под покупателем, добавить товары в корзину, оформить заказ — убедиться, что заказ создан, остатки уменьшены, корзина очищена, заказ виден в «Мои заказы».
2. **Управление заказами:** войти под администратором, открыть вкладку «Заказы», отфильтровать по статусу, изменить статус заказа; убедиться, что новый статус отображается в личном кабинете покупателя.
3. **Каталог:** в публичном каталоге задать диапазон цены, включить «Только в наличии», переключить сортировку, пролистать страницы — убедиться, что параметры сохраняются.
4. **Пользователи:** во вкладке «Пользователи» изменить роль выбранного пользователя; убедиться, что собственную роль изменить нельзя; после применения обновлённого триггера проверить запись в журнале аудита.

Доработки считаются принятыми, если перечисленные сценарии выполняются корректно и существующий функционал не нарушен.
